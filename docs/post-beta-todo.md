## 🏎 P0: 核心缺陷修复与基建替换
### 1. 邀请码并发防重放及落库问题修复
- **当前现象描述**：邀请码用过一次本应宣告消耗，目前可反复无尽注册并不会失效。
- **问题技术根源**：强烈怀疑因架构切为 Postgres JSONB 单字段覆盖机制后，前端频繁调用更新和新用户写入时，`consumeInvite` 函数对于 `invite.usedBy` 的修改状态没能被完整带入最新的数据库状态写锁（或被覆盖）。
- **文件检查指针**：
  - Backend 入口: @[backend/src/auth/routes.ts]
  - 核心状态方法: @[backend/src/auth/invite.js]（检查 `consumeInvite(db, code, userId)` 后，是否确实挂载回了 `saveDb(db)` 的主干引用上。）
  - Db 更新落库机制: @[backend/src/db.ts] `savePostgresDb()`。

### 2. 真实验证邮件投递基建集成
- **当前现象描述**：内测目前仍将邮件强制打印在本地 Bash 控制台中。
- **任务目标**：需配合产品团队采购真实云端投递 API。
- **文件检查指针**：
  - 更新 @[backend/.env] 写入 `EMAIL_PROVIDER=resend` 以及 `RESEND_API_KEY=xxx`。
  - 测配路由代码: @[backend/src/auth/routes.ts] (观察发送结果能否抛出第三方 Error)。

## 📍 P3: 泛地理定位引擎与动态坐标筛选升级
### 5.1 从静态列表迁跃至动态地理定位 API (LBS 服务)
- **当前瓶颈**：根据目前 CTO 设计的结构 @[backend/src/universities.ts]，目前的“约会安全咖啡厅地点”全靠人工硬编码（如：固定了 3 个咖啡厅名字），对于两个任意陌生交大校区，匹配到的地点可能是“极其反人类的不方便通勤”。
- **任务目标**：
  - a. 废弃或升级 @[backend/src/universities.ts] 中的硬编码安全区机制。
  - b. 针对匹配后端 `workflowRouter` 计算地点时，接入比如 **Google Maps API (Places API)** 来生成一个距双方最均衡（Midpoint）的动态商区。
  - c. 用户个人资料面板应新增“我常驻的微型活动热图及通勤范围选择”，从而提高 Match 分数和推荐成行率。
